# Codegen feature plan

Self-contained plan for adding ABI → TypeScript-module codegen to
`abi-to-zod`. Designed to be picked up in a fresh context window.

## Goal

Take an Ethereum ABI JSON file and emit a standalone TypeScript module
containing zod schemas (one per function). The emitted module imports
`zod` and nothing else from `abi-to-zod` — it's a self-contained artefact
the user checks into their own repo and customises.

Why: build-time schemas, edit-in-place customisation (`.describe()`,
custom refinements, error maps), real tree-shaking, precise `z.infer`
types, no library at runtime in consumer code.

The existing `abiToZod` / `abiFunctionToZod` runtime path stays — useful
for dynamic ABIs (a UI taking ABI input at runtime). Codegen is the
build-time companion.

## Public surface

### CLI

```
npx abi-to-zod <input-abi.json> [output.ts]
```

If `output.ts` is omitted, write to stdout. No subcommands, no flags. The
binary is wired via `package.json#bin`.

### Programmatic

```ts
import { generate } from 'abi-to-zod/codegen';

const source: string = generate(abi);  // abi is a parsed ABI array
```

Returns the generated TS source as a string. The caller is responsible
for writing it. The CLI is a thin wrapper.

### Sub-path export

Codegen lives at the `./codegen` sub-path so consumers can import it
without bundling the runtime path (and vice versa). Update
`package.json#exports` accordingly:

```json
"exports": {
  ".": { "types": "./dist/index.d.ts", "import": "./dist/index.js" },
  "./codegen": { "types": "./dist/codegen.d.ts", "import": "./dist/codegen.js" }
}
```

## Output format (example: ERC20)

```ts
// Generated by abi-to-zod v0.0.0 from ERC20.json
// Do not edit manually -- regenerate from source.

import { z } from 'zod';

// === Primitive schemas (edit once to customise all uses) ===

const ADDRESS = z.string().regex(/^0x[0-9a-fA-F]{40}$/).transform(v => v as `0x${string}`);
const UINT256 = z.string().regex(/^\d+$/).transform(BigInt).refine(n => n <= (1n << 256n) - 1n);

// === Function schemas ===

// transfer(address,uint256)
export const transferSchema = z.tuple([
  /* to: address */ ADDRESS,
  /* value: uint256 */ UINT256,
]);

// balanceOf(address)
export const balanceOfSchema = z.tuple([
  /* owner: address */ ADDRESS,
]);

// === Lookup barrel (mirrors abiToZod) ===

export const schemas = {
  transfer: transferSchema,
  'transfer(address,uint256)': transferSchema,
  balanceOf: balanceOfSchema,
  'balanceOf(address)': balanceOfSchema,
} as const;
```

### Output rules (locked)

- **Header:** generator version, source file basename, "do not edit".
- **Imports:** `import { z } from 'zod'` only. No `abi-to-zod` runtime import.
- **Primitive consts:** hoisted to the top of the file. One per primitive
  variant that's actually used in the ABI. Names: `ADDRESS`, `BOOL`,
  `STRING`, `BYTES`, `UINT256`, `INT64`, `BYTES32`, etc. Sorted by name.
- **Function schemas:** named `<name>Schema` when the function name is
  unambiguous; **omitted entirely for overloaded functions** (those only
  appear in the barrel). Sorted by canonical signature.
- **Tuples:** rendered inline. No struct-shape dedup in v1.
- **Barrel:** `export const schemas = { ... } as const;` — keys are the
  function name (unambiguous functions only) and the full canonical
  signature (always). The `as const` is load-bearing for precise types.
- **Indent:** 2-space, hardcoded.
- **Determinism:** functions sorted by canonical signature; primitives
  sorted by const name. Same ABI → identical bytes on every run.

### Overload handling

Unambiguous:
```ts
export const transferSchema = z.tuple([...]);
```
+ `schemas.transfer` and `schemas['transfer(address,uint256)']` in the barrel.

Overloaded (`safeTransferFrom` has two arities in ERC-721):
- **No per-function export.**
- Barrel only:
  ```ts
  'safeTransferFrom(address,address,uint256)': /* z.tuple([...]) */,
  'safeTransferFrom(address,address,uint256,bytes)': /* z.tuple([...]) */,
  ```
- The plain name `safeTransferFrom` is NOT a key in the barrel (would be
  ambiguous to consume).

Rationale: avoids inventing a mangling convention (no `selector_<hex>` or
`argTypeSuffix`). Users with overloads access via signature key. viem
doesn't generate per-function exports at all, so there's nothing to
copy; the barrel-only approach is internally consistent with how
`abiToZod` already disambiguates.

## Implementation phases

Each phase is committable independently. Each must pass typecheck +
existing tests before moving on.

### Phase A — share primitive-source rendering with primitive-schema

Refactor `src/primitives.ts` so that `primitiveSchema` and a new
`primitiveSource` go through a single dispatch:

```ts
function dispatchPrimitive<T>(base: string, handlers: {
  uint: (bits: number) => T;
  int: (bits: number) => T;
  bytes: () => T;
  bytesN: (n: number) => T;
  address: () => T;
  bool: () => T;
  string: () => T;
}): T { ... }

export function primitiveSchema(base: string): z.ZodType { ... }
export function primitiveSource(base: string): string { ... }
```

Important: call `dispatchPrimitive<z.ZodType>(...)` with explicit type
arg in `primitiveSchema` — TS otherwise infers `T` from the first
handler and the other variants' return types fail to unify. (Hit this
in the reverted spike on `feat/initial-implementation`.)

Add `renderSchemaSource(param, indent?)` and `renderTupleSource(params,
indent?)` to `src/build.ts`, mirroring the recursion in `buildSchema`.
Same path traversal — only the leaf and join behaviour differs (string
concat vs zod wrappers).

Also collect the set of *used primitives* during render — needed by the
codegen module to emit only the consts it references. Either:

- A separate pass over the ABI (parseType + walk components), or
- Have the renderer return both a `source` string and a `Set<string>`
  of primitive names used.

The two-pass approach is simpler; do that.

**Const-aware rendering:** the rendering helpers should accept a
"primitive name resolver" — given a Solidity base type, return the const
identifier to use (`ADDRESS`, `UINT256`, etc.) instead of inlining. This
keeps the rendering generic; the codegen pass supplies the resolver.

#### Phase A tests

- For each primitive variant, `primitiveSource(base)` returns a string
  whose runtime behaviour matches `primitiveSchema(base)`. Concretely:
  `eval(`(() => ${source})()`)` produces a zod schema that accepts
  exactly the same inputs and yields equivalent outputs.
- `renderTupleSource` handles nested tuples, arrays, fixed arrays
  matching `buildSchema`.

### Phase B — codegen module

`src/codegen.ts`:

```ts
export function generate(abi: Abi, sourceName?: string): string;
```

`sourceName` is used in the header comment (e.g., the filename); falls
back to `'(unnamed)'`.

Algorithm:

1. Filter ABI to function entries (reuse the strict filter from
   `abi.ts`; throw on malformed function entries — matches runtime
   behaviour).
2. For each function, collect:
   - canonical signature (reuse `canonicalSignature` from `abi.ts`)
   - whether the name is overloaded (more than one function with that name)
   - set of primitive constants used
3. Walk all params to collect the union of primitive constants needed
   across the whole file.
4. Emit:
   - Header comment (generator version from `package.json`, source name).
   - `import { z } from 'zod';`
   - Primitive consts, sorted by name.
   - Per-function exports, sorted by canonical signature, skipping
     overloaded ones.
   - Barrel object, keys sorted: name keys (unambiguous only), then
     signature keys.

#### Phase B tests

- **Equivalence test** (strongest): for every fixture, eval the
  generated source (via dynamic `tsx` import in a temp dir, or via
  `eval` with `new Function` wrapping). Parse placeholder args with both
  the runtime schema (`abiFunctionToZod`) and the generated schema.
  Deep-equal. This catches any drift between the two paths.
- Snapshot test on 1-2 small fixtures (ERC20, ArbInfo) for human-readable diff coverage.

### Phase C — CLI

`src/cli.ts`:

```ts
#!/usr/bin/env node
import { readFileSync, writeFileSync } from 'node:fs';
import { basename } from 'node:path';
import { generate } from './codegen.js';

const [, , input, output] = process.argv;
if (!input) {
  console.error('Usage: abi-to-zod <input-abi.json> [output.ts]');
  process.exit(1);
}
const abi = JSON.parse(readFileSync(input, 'utf8'));
const src = generate(abi, basename(input));
if (output) writeFileSync(output, src);
else process.stdout.write(src);
```

Package wiring:

- `package.json#bin`: `{ "abi-to-zod": "./dist/cli.js" }`.
- Build emits with `#!/usr/bin/env node` shebang preserved by tsc (it
  doesn't strip these).
- Need an executable bit on the built file. tsc doesn't set this — add
  a post-build script: `chmod +x dist/cli.js`. Or wire via npm publish
  prep. For now, a `postbuild` script is simplest.

#### Phase C tests

- Spawn `node dist/cli.js path/to/abi.json` on a fixture, capture
  stdout, assert it matches `generate(abi)` from the programmatic API.
- Spawn with an output path, assert the file exists and matches.

### Phase D — golden fixtures

For each of the 24 fixtures, commit a pre-generated output file:

```
test/fixtures-generated/
  arbitrum/l1/Bridge.ts
  arbitrum/l1/EdgeChallengeManager.ts
  ...
  erc/ERC20.ts
  ...
  mainnet/Seaport.ts
```

A test regenerates each one in memory and compares byte-for-byte against
the committed version. Catches accidental generator drift across PRs.

To regenerate (when intended): a script like `scripts/regenerate-golden.mjs`
that walks `test/fixtures/**/*.json` and writes the corresponding
`test/fixtures-generated/<same-path>.ts`. Run it manually when generator
output changes intentionally; committed diff goes in the same PR as the
generator change.

## Out of scope for this iteration

- Events, errors, constructor, fallback, receive entries.
- Output (return-value) schemas.
- `.describe()` auto-injection from `internalType`.
- Inlining vs hoisting flag (`--no-consts`).
- Indent option.
- Struct-shape dedup (identical tuple shapes used in multiple functions
  stay inlined).
- Watch mode (regenerate on ABI file changes).
- Multi-ABI input (one file per call; users can wrap in their own loop).

## Open questions resolved

- **Subcommand:** none. Just `npx abi-to-zod <input> [output]`.
- **Flags:** none in v1.
- **Indent:** 2-space hardcoded.
- **Golden file scope:** all 24 fixtures.
- **Overload naming:** barrel-only by signature; no per-function export
  for overloaded names.
